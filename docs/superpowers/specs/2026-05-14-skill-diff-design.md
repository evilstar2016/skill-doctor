# Design: `skill-doctor diff` Command

**Date:** 2026-05-14  
**Status:** Approved

## Overview

Add a `diff` command that performs a systematic semantic comparison of two local skills, helping users decide which one to use.

```
skill-doctor diff <skill-a> <skill-b> [--report path.html]
```

Default output: terminal. With `--report`: HTML file (same visual style as `audit --report`).

## Architecture

### New files

```
src/diff/
  runDiff.ts         — main orchestration
  analyzeDiff.ts     — LLM analysis call
  types.ts           — DiffResult / SkillProfile types

src/render/
  renderDiff.ts      — terminal renderer
  renderDiffReport.ts — HTML renderer
```

### Reused existing modules

- `src/parsing/` — parse SKILL.md, extract structured fields
- `src/discovery/` — locate skill by name
- `src/render/renderAuditReport.ts` — HTML template style reference
- `src/explain/` — LLM call wrapper

### Data flow

```
CLI args
  → discover both skill paths (src/discovery)
  → parse each SKILL.md → SkillProfile (rule-based, no LLM)
  → LLM call with both profiles → structured DiffResult.analysis
  → renderDiff (terminal) or renderDiffReport (HTML)
```

## Data Structures

```ts
interface SkillProfile {
  name: string;
  description: string;
  whenToUse: string;
  triggers: string[];
  checklistItems: string[];
  rawContent: string;        // full text passed to LLM
}

interface DiffResult {
  skillA: SkillProfile;
  skillB: SkillProfile;
  analysis: {
    prosConsA: { pros: string[]; cons: string[] };
    prosConsB: { pros: string[]; cons: string[] };
    triggerComparison: string;
    coverageOverlap: string[];
    coverageOnlyA: string[];
    coverageOnlyB: string[];
    situationalAdvice: Array<{
      condition: string;             // "如果你是…"
      recommendation: "A" | "B" | "either";
      reason: string;
    }>;
  };
}
```

`SkillProfile` is extracted by rule-based parsing (fast, no LLM dependency).  
`DiffResult.analysis` is generated by a single structured LLM call.

## Analysis Dimensions

1. **Pros/cons** — strengths and weaknesses of each skill
2. **Use cases & audience** — when and who each skill suits
3. **Trigger condition comparison** — how their when-to-use / trigger fields differ
4. **Functional coverage** — overlap, exclusive-to-A, exclusive-to-B
5. **Situational advice** — conditional recommendations ("if you are X, use Y")

## HTML Report

Same visual structure as `audit --report`: header summary card + collapsible dimension sections + highlighted conclusion block.

## Error Handling

| Scenario | Behavior |
|----------|----------|
| Skill name not found | Exit with error, list available names |
| Both names are the same | Exit with error, require two distinct skills |
| SKILL.md missing key fields | Fall back to empty string, mark as "not found" in output |
| LLM call fails / times out | Render rule-based sections, mark LLM dimensions as "analysis unavailable" |
| `--report` path not writable | Error on file write, terminal output still renders |
